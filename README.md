# operating-platforms-snhu
A portfolio of projects completed and progress made throughout CS230 at SNHU<br><br>
**Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?<br>**
The Gaming Room wanted to expand their Android game, Draw It or Lose It, into a web-based, multi-platform application. They needed software that could support multiple users across different devices while ensuring consistent gameplay, unique game and team names, and only one active game instance in memory at a time.<br><br>
**What did you do particularly well in developing this documentation?<br>**
I clearly defined the system architecture and applied object-oriented principles such as inheritance, encapsulation, and the Singleton pattern to meet the client’s requirement of a single active game instance. The UML diagram and domain model effectively mapped out relationships between Game, Team, and Player classes.<br><br>
**What about the process of working through a design document did you find helpful when developing the code?<br>**
Working through the design document helped me organize the system before writing code. It clarified class relationships, constraints, and architectural decisions early, which reduced confusion and made implementation more structured and efficient.<br><br>
**If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?<br>**
If I could revise one part, I would improve the evaluation section by making the technical comparisons more concise and directly tied to the client’s business goals. I would also include clearer justification connecting system architecture decisions to scalability and performance outcomes.<br><br>
**How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?**<br>
I translated the client’s need for consistency and scalability into a centralized server architecture with controlled memory management and unique identifiers. Considering user needs is critical because the software must solve real problems efficiently, remain intuitive, and scale as usage grows.<br><br>
**How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?**<br>
I approached the design by first analyzing requirements, identifying constraints, and modeling the domain with UML before selecting architecture and platform recommendations. In the future, I would continue using requirement analysis, object modeling, design patterns, and scalability planning to guide similar software projects.
